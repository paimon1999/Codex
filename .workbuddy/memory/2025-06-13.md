## 模型显示名称注入功能

### 问题描述
- Codex UI 显示"自定义"而不是模型名称
- `model_provider` 字段只能是 "custom"，Codex 硬编码只识别这个值
- 用户希望显示当前使用的模型名称（如 "mimo-v2.5-pro"）

### 解决方案
通过注入脚本动态替换 UI 中的"自定义"/"custom"为当前模型名称。

### 实现细节

#### 1. Helper Server 添加 `/config` API 端点
**文件**: `launcher.rs`

```rust
if path == "/config" && method == "GET" {
    // 读取 config.toml
    // 返回 JSON: { model, modelProvider, displayName }
}
```

**返回数据**:
- `model`: 当前使用的模型名称（如 "mimo-v2.5-pro"）
- `modelProvider`: 当前使用的供应商 ID（如 "custom"）
- `displayName`: 用于显示的名称（优先使用 model 字段）

#### 2. 注入脚本添加动态替换逻辑
**文件**: `renderer-inject.js`

```javascript
// 从 Helper Server 获取配置

}

// 替换 DOM 中的"自定义"/"custom"
```

## 启动功能修复

### 问题分析
1. Manager 没有"启动 Codex"按钮，只有"启动 Ucodex"（带注入）
2. `spawn_silent_launcher` 只调用 `command.spawn()` 就返回 `Ok(())`，不验证进程是否真正启动
   - launcher 可能立即崩溃（找不到 Codex App、端口占用等），但 Manager 仍显示"启动成功"

### 修复方案

#### 后端修改（commands.rs）
1. **添加 `launch_codex_app_only` 命令**：直接启动原版 Codex App（不注入）
   - macOS: 使用 `open` 命令启动 .app bundle，附加 `--remote-debugging-port` 参数
   - Windows/Linux: 直接运行可执行文件
   - 等待 1.5 秒后检查 Codex 进程是否出现
   - 如果未检测到进程，返回错误

2. **修改 `spawn_codex_plus_launch`**：启动后验证
   - 检查 launcher 二进制是否存在
   - spawn 后等待 800ms，用 `kill -0` 检查进程是否存活
   - 同时检查 Codex 进程是否已出现
   - 如果 launcher 已退出且 Codex 未启动 → 返回错误（含具体原因）

3. **修改 `spawn_silent_launcher`**：返回 PID 而非 `()`
   - 返回 `anyhow::Result<u32>` 而非 `anyhow::Result<()>`
   - 前置检查 launcher 二进制是否存在

#### 前端修改（App.tsx）
1. 添加 `launchCodexAppOnly` 函数
2. 更新 Actions 类型
3. 在概览页面和维护页面添加"启动 Codex（无注入）"按钮

### 关键文件
- `apps/codex-plus-manager/src-tauri/src/commands.rs`: 启动逻辑
- `apps/codex-plus-manager/src-tauri/src/lib.rs`: 命令注册
- `apps/codex-plus-manager/src/App.tsx`: 前端 UI

## Helper Server 端口冲突修复

### 根本原因
- Manager 启动时内嵌 Helper Server 绑定 127.0.0.1:57321
- 用户点击"启动 Ucodex"时，launcher 也尝试绑定同一端口 → AddrInUse → 启动失败
- launcher 没有 `--skip-helper` 参数，无法跳过 Helper 启动

### 修复方案（launcher.rs）
修改 `DefaultLaunchHooks::start_helper` 方法：
1. 尝试 `TcpListener::bind` helper_port
2. 如果成功 → 正常启动 Helper Server
3. 如果 AddrInUse → 发 HTTP GET 到 `http://127.0.0.1:{port}/backend/status` 检查是否已有 Helper
   - 200 OK → 已有 Helper Server，跳过启动，记录 `helper.reused` 日志，返回 Ok(())
   - 其他 → 端口被其他程序占用，返回原错误

### 关键代码
```rust
Err(e) if e.kind() == std::io::ErrorKind::AddrInUse => {
    let url = format!("http://127.0.0.1:{helper_port}/backend/status");
    let client = reqwest::Client::builder()
        .timeout(std::time::Duration::from_secs(2))
        .build()
        .unwrap_or_default();
    match client.get(&url).send().await {
        Ok(resp) if resp.status().is_success() => {
            // 已有 Helper Server 在运行，跳过
            Ok(())
        }
        _ => Err(anyhow::anyhow!("port in use by another process"))
    }
}
```

### 关键文件
- `crates/codex-plus-core/src/launcher.rs`: `start_helper` 方法
- `apps/codex-plus-manager/src-tauri/src/commands.rs`: 启动验证 + stderr 捕获

## Codex 无调试端口重启修复

### 问题
日志显示 `helper.reused` 成功后，`ensure_injection` 反复失败（120次）：
```
CDP connection refused on port 9229
```
原因：Codex 已在运行（不带调试端口），`open -a` 只激活已有实例，不传新参数。

### 修复
在 `launch_codex` macOS 路径中：
1. 检查 `is_macos_app_running` + `cdp_listening(debug_port)`
2. 如果 Codex 运行中且 CDP 可用 → 直接使用
3. 如果 Codex 运行中但 CDP 不可用 → 用 osascript 退出 Codex，等待完全退出后重新启动
4. 如果 Codex 未运行 → 正常启动

### 关键代码（launcher.rs launch_codex 方法）
```rust
let already_running = is_macos_app_running(app_dir).await;
let cdp_available = crate::watcher::cdp_listening(debug_port);
if already_running && !cdp_available {
    // osascript quit + 等待退出 + 重新启动
}
```
