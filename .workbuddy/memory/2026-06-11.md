## 修复：自定义模型注入时序问题

**根因**：`patchCodexModelWhitelist()` 的执行时序竞争——首次调用时 model catalog 尚未加载，触发异步加载后虽然会再次调用 patch，但 Statsig 可能已缓存了 config 并完成 UI 渲染，React 组件不会重新渲染。

**修复**（`assets/inject/renderer-inject.js`）：
- 添加 `scheduleModelWhitelistRetry()` 函数，模型注入成功后启动周期性重试
- 每 2 秒重新执行 `patchStatsigModelWhitelist()` + `patchReactModelState()`
- 最多重试 15 次（30 秒），之后自动停止
- 如果用户禁用了 modelWhitelistUnlock 或 models 为空，提前停止
- Statsig 的 `getDynamicConfig` wrapper 已经是动态检查 `codexPlusModelNames()` 的，重试时会正确注入新模型

## 二次修复：上游协议方法名变更

**根因**：上游 Codex App 已将模型列表协议从 `list-models-for-host` 改为 `model/list`，但注入脚本的 `patchAppServerModelResult` 仍在检查旧方法名，导致 App Server request patch 路径无法拦截模型响应。

**修复**（`assets/inject/renderer-inject.js` 第 4222 行）：
- `patchAppServerModelResult` 方法检查从 `method !== "list-models-for-host"` 改为 `method !== "list-models-for-host" && method !== "model/list"`
- MCP 路径（`patchAppServerModelMessages`）已正确拦截 `model/list`，但 App Server request patch 路径需要同步更新

**上游源码参考**：
- `codex-rs/app-server-protocol/src/protocol/common.rs:840` — `ModelList => "model/list"`
- `codex-rs/app-server/src/models.rs:12` — `supported_models()` 使用 `thread_manager.list_models()` 获取模型
- `codex-rs/models-manager/src/manager.rs:82` — `list_models()` 从远程 `/models` 端点获取模型并缓存

## Mimo 401 错误修复

用户 Codex 发请求到 `api.openai.com/v1/responses` 返回 401，因为 config.toml 只有 `model = "mimo-v2.5-pro"` 没有 `model_provider` 和 provider 表。

**修复**（`~/.codex/config.toml`）：
- 添加 `model_provider = "mimo"`
- 添加 `[model_providers.mimo]` 表，`base_url` 指向本地协议代理 `http://127.0.0.1:57321/v1`
- 添加 `codex_plus_chat_base_url = "https://token-plan-cn.xiaomimimo.com/v1"` 作为 Mimo Token Plan 上游地址
- `experimental_bearer_token` 写入 Mimo API key

**关键知识**：Mimo 不兼容 Responses API，需要通过 Ucodex 协议代理做 Responses→ChatCompletions 转换。Token Plan key（tp- 开头）对应 `token-plan-cn.xiaomimimo.com/v1`。
